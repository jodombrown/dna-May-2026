## Executive Summary (preliminary)

The codebase is structurally healthy on the foundations that matter most for an alpha:
RLS is enabled on 100% of public tables (248/248), all three messaging auto-create
triggers ship at the database layer, and the DIA naming compliance is clean (zero
`Assistant` references in DIA paths). The Five C's open-modules invariant holds and
the composer ships 5 of 6 modes.

The biggest risks surfaced in Turn 1 are not architectural - they are drift between
spec and code:

1. **Reactions drift (CRITICAL):** `src/types/reactions.ts` defines `'like' | 'love' | 'celebrate' | 'insightful' | 'support' | 'curious'` instead of the canonical five (Asante, Inspired, Let's Build, Powerful, Insightful). This is a binding spec violation.
2. **TypeScript strict drift:** 143 `: any` annotations + 301 `as any` casts = 444 untyped surfaces against a "zero `any`" project rule. Most are concentrated in releases/admin/composer surfaces, not core C's.
3. **`space_type` is unconstrained:** the DB column is `text` with no enum. The spec mandates exactly `project | working_group | initiative | program`. Code can write any string.
4. **Em-dashes in 20+ source files:** binding constraint violation, mostly in comments and a few user-visible strings.

None of these block alpha; all four should land in a single hardening sprint before
broader rollout.

### Key finding: dual-write contradiction

`create_space_messaging_channel(p_space_id)` is **already written defensively to look up the space in `spaces` first, then fall back to `collaboration_spaces`**. The platform's own messaging RPC treats `spaces` as primary and `collaboration_spaces` as secondary. This is the strongest architectural signal in the codebase.

Conversely, the universal Post Composer (`composerService.ts:351`) currently writes to `collaboration_spaces` - meaning every space created through the canonical creation surface is invisible to `space_members`, `space_tasks`, `initiatives`, `space_roles`, `contribution_needs`, `contribution_offers`, and the `nudges` engine. **This is the bug that gates the Collaborate rebuild.**

### Recommendation

**Canonical: `spaces`.** Deciding factor: 11 FK children vs 1, plus the full Collaborate domain model (tasks, initiatives, roles, nudges, health) hangs off `spaces`. `collaboration_spaces` only owns `posts.space_id` and a parallel membership table that duplicates `space_members` with renamed roles. Migrating `posts.space_id` to point at `spaces` is one FK swap; migrating tasks/initiatives/roles/needs/offers/nudges/health off `spaces` would touch ~11 tables and rewrite the rebuild surface area.
