# D054 Migration Plan — v0.0 Three Identity Layers + DNA × Place

**Status:** PLAN ONLY — no migrations applied, no application code modified.
**Cycle:** v0.0 (BD009: mobile-first PWA on the existing `jodombrown/dna` repo and the existing Supabase project at `https://ybhssuehmfnxrzneobok.supabase.co`).
**Companion docs:** [`../current-schema.md`](../current-schema.md), [`../current-flows-audit.md`](../current-flows-audit.md).

D054 introduces three identity layers on top of the existing `profiles` table:
1. **Profile layer** — extend `profiles` with role + place columns.
2. **Role layer** — typed identity role: `returnee`, `anchor`, `ally`, `exploring`. `exploring` is the soft option per BD008 / D054.
3. **Affirmation layer** — new `affirmations` table for a one-time witnessed declaration of commitment to The Return. Ally affirmations require a witness inside the kinship line.
4. **DNA × Place** — `continent` (text) + `country` (ISO 3166-1 alpha-3) on `profiles`. Hierarchy extends to region/state/city later per D029/D030.

The plan also addresses BD008 forced-onboarding for pre-migration users.

---

## Guiding constraints

- **Extend, never replace.** `profiles` already has ~180 columns including overlapping country/location fields. We add new canonical D054 columns and leave the legacy columns alone.
- **No data loss.** Existing rows get nullable new columns (or a soft `'exploring'` default for role per BD008); we never break a working session.
- **Backward-compatible at the API surface.** Existing `select('*')` callers (`AuthContext.tsx:91`, `useProfile.ts`, etc.) will start seeing the new fields automatically; no consumer breaks.
- **Single integration point for the forced-declaration gate.** `src/components/auth/OnboardingGuard.tsx` already wraps ~70 `/dna/*` routes and is the existing place to widen the check.
- **RBAC unchanged.** `user_roles.app_role` (`user|moderator|admin`) is permissions; the new `profiles.role` is identity. They coexist.

---

## Migration order (logical, not file-numbered)

This plan describes four migrations applied in order:

| # | What | Why ordered here |
|---|---|---|
| M1 | Create enum `dna_identity_role` | Type must exist before the column references it |
| M2 | Extend `profiles` (role, continent, country) + soft default | Other tables/RPCs can then key off these columns |
| M3 | Create `affirmations` table + RLS + ally-witness trigger | Depends on `dna_identity_role` (witness role check) and on `profiles.continent/country` (kinship-line check) |
| M4 | (No SQL — RLS amendments to `profiles` if/when role-aware visibility is added later) | Out of scope for v0.0 read paths |

File-numbering convention to follow existing repo style: timestamp-prefixed SQL in `supabase/migrations/`. Suggested skeleton names:

- `<ts>_d054_create_identity_role_enum.sql`
- `<ts>_d054_extend_profiles_role_place.sql`
- `<ts>_d054_create_affirmations.sql`

## M1 — Create `dna_identity_role` enum

### Exact SQL

```sql
CREATE TYPE public.dna_identity_role AS ENUM (
  'returnee',
  'anchor',
  'ally',
  'exploring'
);
```

### Rationale

- D054 defines exactly four values; an enum gives DB-level integrity and lets RLS / triggers / views constrain on the type.
- The name is **`dna_identity_role`**, deliberately distinct from the existing `public.app_role` enum (`user|moderator|admin`) which is RBAC. Two different concepts → two different types.
- We do **not** reuse `profiles.user_type` (text default `'member'`), `profiles.user_role` (text), `profiles.roles text[]`, or `profiles.diaspora_status` (text, free-form using values like `'returnee'` and `'ally'` via `CONNECTION_TYPE_OPTIONS` in `src/data/profileOptions.ts:50-58`). Those fields stay; they capture different facets (ancestry, account class, free-form tagging) and will be reconciled later. The audit doc §1 covers the overlap.

### Default value strategy

Enums have no default; the default lives on the column. See M2.

### Backward-compatibility implications

None directly. Creating a type is free.

### RLS implications

None directly.

## M2 — Extend `profiles` with `role`, `continent`, `country`

### Exact SQL

```sql
-- Role layer
ALTER TABLE public.profiles
  ADD COLUMN role public.dna_identity_role NOT NULL DEFAULT 'exploring';

-- DNA × Place
ALTER TABLE public.profiles
  ADD COLUMN continent text NULL,
  ADD COLUMN country   char(3) NULL;

-- ISO 3166-1 alpha-3 shape guard (uppercase three letters).
-- Validation against the live ISO list is enforced at the application layer
-- (CountrySelect already feeds a vetted source); a soft regex CHECK keeps
-- garbage out of the column.
ALTER TABLE public.profiles
  ADD CONSTRAINT profiles_country_iso3_format
  CHECK (country IS NULL OR country ~ '^[A-Z]{3}$');

-- D054 v0.0 lists 5 continents per The Return geography; we don't enum-lock
-- this in v0.0 because D029/D030 will expand the place hierarchy. Soft CHECK only.
ALTER TABLE public.profiles
  ADD CONSTRAINT profiles_continent_known
  CHECK (continent IS NULL OR continent IN (
    'Africa', 'Asia', 'Europe', 'North America', 'South America', 'Oceania', 'Antarctica'
  ));

-- Indexes to support feed/discover filters by role and place.
CREATE INDEX IF NOT EXISTS profiles_role_idx       ON public.profiles (role);
CREATE INDEX IF NOT EXISTS profiles_continent_idx  ON public.profiles (continent);
CREATE INDEX IF NOT EXISTS profiles_country_idx    ON public.profiles (country);

-- Surface the new columns on the public_profiles view so feed/discover
-- RPCs can keep using the view consistently. (View is read by many
-- relationships per types.ts; see ../current-schema.md §25.)
DROP VIEW IF EXISTS public.public_profiles;
-- The actual CREATE VIEW DDL must mirror the existing definition with the
-- three new columns added. This is a placeholder for the real DDL that will
-- be authored when the migration is written, by reading the live view body
-- via pg_get_viewdef('public_profiles'::regclass).
-- CREATE VIEW public.public_profiles AS
--   SELECT ..., role, continent, country
--   FROM public.profiles
--   WHERE is_public = true;
```

### Rationale (per statement)

- **`role NOT NULL DEFAULT 'exploring'`** — BD008 mandates a soft "Still exploring" option as the no-op identity. Defaulting to `'exploring'` means every existing row immediately has a valid role; we never break any RLS predicate that might reference `role` later. The forced-onboarding gate (see "Existing-user migration flow" below) treats `'exploring'` as "not yet declared" and prompts on next login, BUT does not block read access.
- **`continent text NULL` / `country char(3) NULL`** — v0.0 minimum per D054. They start NULL because there is no safe automatic mapping from the existing zoo of country fields (`current_country`, `current_country_code`, `current_country_id`, `country_of_origin`, ...) without policy decisions about which one is canonical. The forced-onboarding gate makes the user supply them; we resist guessing.
- **`char(3)` + regex CHECK** — keeps the storage tight and matches the existing `countries.country_code_iso3` shape (per migration `20251218100001_create_countries_table.sql`). No FK to `countries(country_code_iso3)` in v0.0 — the canonical pivot remains the existing `countries` table by `id`; the new `country` column is a denormalized convenience for feed/profile rendering. A later cleanup migration can add the FK once data is fully populated.
- **Continent CHECK (soft)** — not an enum because D029/D030 will widen the place hierarchy and we don't want to ALTER TYPE under load later.
- **Indexes** — feed tabs and discover filters will be common predicates: `role`, `continent`, `country`. Single-column btree indexes are cheap.
- **`public_profiles` view rebuild** — many tables FK to the view (per `types.ts`), and feed RPCs join through it. The new columns must be visible there too. (The view's CREATE statement isn't in `supabase/migrations/` in a discoverable form — it must be reconstructed from `pg_get_viewdef('public.public_profiles'::regclass)` at migration-authoring time.)

### Default value strategy for existing rows

- `role`: backfilled to **`'exploring'`** automatically by `DEFAULT 'exploring'` on `ADD COLUMN NOT NULL`. No backfill SQL needed.
- `continent`: stays **NULL**. The forced-onboarding gate will require the user to set it on next login. We do NOT automap from `current_country_id` → `countries.region.continent_id → continents.name` in v0.0 because:
  - The chain has gaps (many profiles never set `current_country_id`).
  - Picking either current-residence or country-of-origin to derive "continent of identity" is a product call, not a migration call.
- `country`: stays **NULL** for the same reason. (Optional future cleanup migration: backfill from `countries.country_code_iso3` joined on `current_country_id` for rows that have it; out of scope for v0.0.)

### Backward-compatibility implications

**For `/auth` flow:**
- `src/contexts/AuthContext.tsx:91` does `select('*').eq('id', userId).maybeSingle()` — new fields appear automatically in the in-memory `UserProfile`. The TypeScript interface at lines 9-48 will lag (no `role`/`continent`/`country` typed), but reads still succeed; consumers that don't reference the new fields are unaffected. TypeScript types regenerate from the live DB into `src/integrations/supabase/types.ts` separately and should be re-pulled after migrate.
- `src/pages/Onboarding.tsx:154-188` upserts a subset of profile fields — unaffected.
- `src/pages/Auth.tsx:51-57` selects `first_name, full_name` — unaffected.

**For `/dna/feed` flow:**
- `get_universal_feed` RPC currently SELECTs `prof.username, prof.full_name, prof.avatar_url, prof.headline` only (see `supabase/migrations/20260508192817_0e10c69b-dcac-4f52-a063-ce71327c1353.sql:60-103`). It is **unaffected** by adding columns to `profiles`; it will keep returning the same shape. If we later want to render role badges on cards, we'll need a follow-up migration to redefine the RPC and update `src/lib/feed/mapFeedRow.ts` and `src/types/feed.ts` (called out in the audit doc §6) — **not part of D054 v0.0**.
- `src/components/feed/PostCard.tsx:59`, `CommentDrawer.tsx:105`, `CommentSection.tsx:259`, `SpotlightCard.tsx:40`, `FeedRightSidebar.tsx:46` all do `select('*')` from `profiles` — they will pass through the new fields and ignore them until UI work is added.

**Other consumers that won't notice:**
- `useProfileCompletion.ts` — unchanged predicates (avatar/headline/location/bio/skills/sectors). It will grow new predicates (`role`, `dna_place`) in a follow-up app-code task, NOT in this migration.

**Potential gotchas:**
- Any code path that does `INSERT INTO profiles (...)` without listing columns explicitly (`.insert([{ id, email, ... }])`) is fine — the `DEFAULT 'exploring'` covers `role`. Spot-checked: `handle_new_user()` trigger inserts `(id, email, full_name)` — covered.
- Anything that does `INSERT ... SELECT ... FROM profiles` from another deployment is out of scope (we have one production DB).

### RLS policy implications

**Existing `profiles` policies (per `supabase/migrations/20250628212148-3c308b6c-...sql` and `20250703034619-5b104a26-...sql`):**
- `SELECT USING (is_public = true OR auth.uid() = id)` — unaffected.
- `INSERT WITH CHECK (auth.uid() = id)` — unaffected.
- `UPDATE USING (auth.uid() = id)` — unaffected.

**No new policies required in v0.0.** The new columns inherit the existing policies. If a future iteration wants role-aware visibility (e.g. "anchors can see other anchors' country regardless of `is_public`"), a separate migration adds that. Out of scope for v0.0.

**Self-update of `role`:** A user can self-promote from `'exploring'` → `'returnee' | 'anchor' | 'ally'` via the existing UPDATE policy. **The witness/affirmation constraint is enforced on the `affirmations` table** (M3), not by blocking the `role` update. This is intentional: role is the user's stated identity; affirmation is the witnessed commitment. They can diverge briefly (declared role with no affirmation yet) and the OnboardingGuard catches that. Discussion point with product: should `role = 'ally'` be disallowed without a matching affirmation row? (Recommend: no for v0.0, gate at UI; reconsider when introducing role-locking.)

## M3 — Create `affirmations` table + RLS + ally-witness trigger

### Exact SQL

```sql
-- One-time witnessed declaration of commitment to The Return.
CREATE TABLE public.affirmations (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  profile_id      uuid NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  role_at_affirm  public.dna_identity_role NOT NULL,
  witness_id      uuid NULL REFERENCES public.profiles(id) ON DELETE SET NULL,
  statement       text NULL,                                  -- optional commitment text (≤ 1000 chars)
  affirmed_at     timestamptz NOT NULL DEFAULT now(),
  created_at      timestamptz NOT NULL DEFAULT now(),

  CONSTRAINT affirmations_statement_length CHECK (statement IS NULL OR char_length(statement) <= 1000),
  CONSTRAINT affirmations_witness_not_self CHECK (witness_id IS NULL OR witness_id <> profile_id),
  CONSTRAINT affirmations_role_not_exploring CHECK (role_at_affirm <> 'exploring')
);

-- One affirmation per profile (v0.0: declaration is one-time).
CREATE UNIQUE INDEX affirmations_one_per_profile_idx ON public.affirmations (profile_id);

-- Lookup indexes
CREATE INDEX affirmations_witness_idx ON public.affirmations (witness_id);
CREATE INDEX affirmations_role_idx    ON public.affirmations (role_at_affirm);

ALTER TABLE public.affirmations ENABLE ROW LEVEL SECURITY;

-- Read: visible to the declarer and the witness; otherwise visible only when
-- the declarer's profile is public (so feed/profile chrome can render a badge).
CREATE POLICY "Affirmations are readable by declarer, witness, and on public profiles"
  ON public.affirmations
  FOR SELECT
  USING (
    profile_id = (SELECT auth.uid())
    OR witness_id = (SELECT auth.uid())
    OR EXISTS (
      SELECT 1 FROM public.profiles p
      WHERE p.id = affirmations.profile_id AND p.is_public = true
    )
  );

-- Insert: the declarer creates their own affirmation.
CREATE POLICY "Users can create their own affirmation"
  ON public.affirmations
  FOR INSERT
  WITH CHECK (profile_id = (SELECT auth.uid()));

-- Update: declarer can only edit the statement; witness cannot rewrite the
-- declaration. Enforced by trigger below (RLS UPDATE alone can't constrain
-- which columns are mutated).
CREATE POLICY "Declarer can update their affirmation"
  ON public.affirmations
  FOR UPDATE
  USING (profile_id = (SELECT auth.uid()))
  WITH CHECK (profile_id = (SELECT auth.uid()));

-- No DELETE policy — affirmations are durable; revocation is a future feature.

-- Witness rules:
--   - ally  ⇒ witness REQUIRED, witness.role IN ('returnee', 'anchor'),
--             AND witness shares kinship line with declarer
--             (v0.0 kinship line = same country OR same continent).
--   - returnee | anchor ⇒ witness OPTIONAL (self-attested allowed in v0.0).
CREATE OR REPLACE FUNCTION public.enforce_affirmation_witness_rules()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_declarer public.profiles%ROWTYPE;
  v_witness  public.profiles%ROWTYPE;
BEGIN
  SELECT * INTO v_declarer FROM public.profiles WHERE id = NEW.profile_id;
  IF NOT FOUND THEN
    RAISE EXCEPTION 'declarer profile % not found', NEW.profile_id;
  END IF;

  IF NEW.role_at_affirm = 'ally' THEN
    IF NEW.witness_id IS NULL THEN
      RAISE EXCEPTION 'ally affirmation requires a witness';
    END IF;

    SELECT * INTO v_witness FROM public.profiles WHERE id = NEW.witness_id;
    IF NOT FOUND THEN
      RAISE EXCEPTION 'witness profile % not found', NEW.witness_id;
    END IF;

    IF v_witness.role NOT IN ('returnee', 'anchor') THEN
      RAISE EXCEPTION 'ally witness must be a returnee or anchor (got %)', v_witness.role;
    END IF;

    -- Kinship line: same country (preferred) OR same continent (fallback).
    IF NOT (
      (v_declarer.country   IS NOT NULL AND v_declarer.country   = v_witness.country)
      OR
      (v_declarer.continent IS NOT NULL AND v_declarer.continent = v_witness.continent)
    ) THEN
      RAISE EXCEPTION 'ally witness must be inside the kinship line (shared country or continent)';
    END IF;
  END IF;

  -- Block mutation of role_at_affirm, profile_id, witness_id, affirmed_at after insert.
  IF TG_OP = 'UPDATE' THEN
    IF NEW.profile_id     IS DISTINCT FROM OLD.profile_id
    OR NEW.role_at_affirm IS DISTINCT FROM OLD.role_at_affirm
    OR NEW.witness_id     IS DISTINCT FROM OLD.witness_id
    OR NEW.affirmed_at    IS DISTINCT FROM OLD.affirmed_at
    THEN
      RAISE EXCEPTION 'only the statement field is mutable on an affirmation';
    END IF;
  END IF;

  RETURN NEW;
END;
$$;

CREATE TRIGGER enforce_affirmation_witness_rules_trg
  BEFORE INSERT OR UPDATE ON public.affirmations
  FOR EACH ROW EXECUTE FUNCTION public.enforce_affirmation_witness_rules();
```

### Rationale (per element)

- **Separate table, not a JSONB blob on `profiles`** — affirmation is a typed event with its own integrity rules (witness reference, role-at-time, durability) and benefits from its own RLS surface.
- **`role_at_affirm` snapshot** — captures what the user was when they affirmed. Future changes to `profiles.role` don't retroactively invalidate the affirmation. CHECK rejects `'exploring'` because that's the soft no-op identity; one doesn't affirm exploration.
- **`witness_id` nullable on the table, required by trigger for ally** — returnee/anchor self-attestation is acceptable in v0.0 (we can tighten later). Ally MUST be witnessed.
- **`ON DELETE SET NULL` for witness** — if a witness deletes their account, the affirmation remains valid; we just lose the witness pointer. The constraint check ran at affirm-time, so historical validity is preserved.
- **Unique index `affirmations_one_per_profile_idx`** — declaration is one-time per BD009/D054 wording. Re-affirmation under a new role (e.g. exploring→returnee, then later returnee→ally) is intentionally NOT supported in v0.0; if the product wants that, drop the unique constraint and add a `revoked_at`/`superseded_by` mechanism.
- **`SECURITY DEFINER` on the trigger** — the trigger needs to read other profile rows (witness, declarer) regardless of the calling user's RLS context. We pin `search_path = public` for safety.
- **Kinship line definition (v0.0)** — same `country` (preferred) OR same `continent`. This deliberately uses the new D054 columns, NOT the legacy zoo of country columns. Implication: if the declarer's `continent`/`country` is NULL at affirm-time (because they were grandfathered without yet completing DNA × Place), the ally constraint will fail. **OnboardingGuard MUST sequence DNA × Place before the affirmation step.** Documented below.
- **Mutability lock** — only `statement` is editable post-insert. Anything else would defeat the "one-time witnessed declaration" semantics.

### Default value strategy for existing rows

`affirmations` is a brand-new table; no backfill. Existing rows in `profiles` start with no affirmation. The forced-onboarding gate prompts users who have `role IN ('returnee','anchor','ally')` but no affirmation row.

### Backward-compatibility implications

**For `/auth` flow:** none — auth flow does not touch `affirmations` today. The future OnboardingGuard widening will add a query for `affirmations` (one row per user, by `profile_id`).

**For `/dna/feed` flow:** none — `get_universal_feed` does not join to `affirmations`. A future "Affirmed" badge on feed cards is a follow-up; out of scope.

**Cascade behaviors:**
- `ON DELETE CASCADE` from `profile_id` → if a user deletes their account (which deletes their `profiles` row via the `auth.users` cascade established in `20250628212148-...sql`), their affirmation also disappears. Correct: affirmations are personal.
- `ON DELETE SET NULL` from `witness_id` → see above.

### RLS policy implications

- `affirmations` enables RLS with the three policies above.
- `profiles` RLS is unchanged.
- The trigger function is `SECURITY DEFINER`, which means it can read `profiles` even when the caller's RLS would not. That's fine — we only need to evaluate role/continent/country of two specific rows; no broad scan, no data leak (the trigger returns no data to the caller, only raises or accepts).

## Existing-user migration flow (BD008)

> All `/dna/*` routes are wrapped in `OnboardingGuard` (`src/components/auth/OnboardingGuard.tsx`); see audit doc §2. This section sketches where the new gating logic slots in. **No code is written in this cycle.**

### What needs to gate

A user is "declared" when:
1. `profiles.role IS NOT NULL` and `profiles.role <> 'exploring'`, OR they have explicitly chosen `'exploring'` in the new onboarding step (recorded somewhere — see below).
2. `profiles.continent IS NOT NULL AND profiles.country IS NOT NULL`.
3. If `profiles.role = 'ally'`, then an `affirmations` row exists for them with a valid witness.

(`returnee` and `anchor` may optionally affirm; not required for v0.0 unblocking.)

Pre-migration users have `role = 'exploring'` (from the M2 default) but **have not seen the new step**, so we can't distinguish "explicitly chose exploring" from "never asked". Two options:

- **Option A (recommended):** add a `profiles.role_declared_at timestamptz NULL` column in M2. Gate fires whenever `role_declared_at IS NULL`, regardless of role value. After a user picks (any value, including `'exploring'`), set `role_declared_at = now()`.
- **Option B:** keep schema minimal and use `onboarding_progress jsonb` (already exists on profiles, NOT NULL with default) to record `{ d054_role_declared: true, d054_place_declared: true }`. Cheaper migration, looser typing.

**Recommendation: Option A.** Add to M2:

```sql
ALTER TABLE public.profiles
  ADD COLUMN role_declared_at timestamptz NULL,
  ADD COLUMN place_declared_at timestamptz NULL;
```

These start NULL for everyone (including the rows that get `role = 'exploring'` by default); the onboarding step sets them on submit.

### Where the gate lives (sketch — NOT code in this cycle)

Insertion point: `src/components/auth/OnboardingGuard.tsx`, inside the existing `useEffect` at lines 38-58. After the existing `hasCompletedOnboarding` short-circuit, add:

```ts
// Pseudocode — do not implement in this planning cycle.
const needsRole  = !profile?.role_declared_at;
const needsPlace = !profile?.place_declared_at;
const declaredRole = profile?.role;
const needsAffirmation =
  declaredRole === 'ally' && /* no affirmations row for user */;

if (hasCompletedOnboarding) {
  if (needsRole && location.pathname !== '/onboarding/declare-role') {
    navigate('/onboarding/declare-role', { replace: true });
    return;
  }
  if (needsPlace && location.pathname !== '/onboarding/declare-place') {
    navigate('/onboarding/declare-place', { replace: true });
    return;
  }
  if (needsAffirmation && location.pathname !== '/onboarding/affirm') {
    navigate('/onboarding/affirm', { replace: true });
    return;
  }
}
```

**Order matters:** role → place → affirmation. The affirmation step needs `country`/`continent` to satisfy the kinship-line constraint (M3 trigger).

**The query** at `OnboardingGuard.tsx:23` must widen its projection:

```ts
.select('onboarding_completed_at, username, role, role_declared_at, continent, country, place_declared_at')
```

…plus a separate React Query for `affirmations` keyed on `user.id`.

### Routes to add (new — NOT created in this cycle)

- `/onboarding/declare-role` — new step component, parallel to `DiasporaOriginStep`. Sets `role` and `role_declared_at`. Options: Returnee / Anchor / Ally / **Still exploring** (soft).
- `/onboarding/declare-place` — new step. CountrySelect → emits ISO-3 → writes `continent` and `country` and `place_declared_at`.
- `/onboarding/affirm` — new step, only reachable for declared roles. Includes witness picker (typeahead of profiles where `role IN ('returnee','anchor')` AND shares country/continent with the declarer). Writes the `affirmations` row.

These routes should NOT be wrapped in `OnboardingGuard` (same pattern as `/onboarding`).

### What happens to a user mid-session at the moment migration runs

- They are mid-`/dna/feed`. M2 lands; their `profiles.role` becomes `'exploring'`, `role_declared_at` is `NULL`. Next route change (or tab focus → React Query refetch) triggers `OnboardingGuard`; they land on `/onboarding/declare-role`. No reload of in-memory session needed because `OnboardingGuard` already runs on every navigation.
- They have already chosen `'exploring'` explicitly later — `role_declared_at IS NOT NULL` short-circuits the gate. They are not re-prompted.

### What happens to a brand-new sign-up post-migration

- `InviteSignup` → `handle_new_user()` trigger inserts a profiles row with `role = 'exploring'` (default), `role_declared_at = NULL`, `continent/country = NULL`. They land on `/dna/feed`; `OnboardingGuard` sees `!onboarding_completed_at && !username` and routes them to `/onboarding` (existing flow). The existing 5-step wizard finishes with `onboarding_completed_at = now()`. Then on next navigation, `OnboardingGuard` sees `!role_declared_at` and routes them to `/onboarding/declare-role`.

(Alternative: extend the existing 5-step wizard to include the new steps inline. Cleaner UX but more app-code surgery. Either path is compatible with this migration plan — the migration only provides the schema.)

## Things deliberately NOT in v0.0

These are called out so reviewers can confirm scope:

- **No backfill of `continent`/`country`** from existing `current_country*` columns. Out of scope; deferred.
- **No FK from `profiles.country` to `countries.country_code_iso3`**. The `countries` table has the column (`20251218100001_create_countries_table.sql:12`), but FK linkage is a future cleanup.
- **No deprecation of the legacy country columns** (`current_country`, `current_country_code`, `current_country_id`, `country_of_origin`, etc.). They keep working; no consumer breaks. A later migration can mark them deprecated once `country`/`continent` adoption is verified.
- **No `get_universal_feed` RPC change**. Role badges / country flags on feed cards are a follow-up; the audit doc §6 documents exactly what to redefine when needed.
- **No `app_role` changes**. RBAC stays untouched.
- **No revocation/re-affirmation flow**. Affirmations are one-time; the unique index enforces it. Revocation can be added later via `revoked_at` + index-predicate update.
- **No role-aware visibility changes on `profiles` RLS**. v0.0 keeps the existing public/own predicate.
- **No region/state/city columns on `profiles`** for DNA × Place — D029/D030 will extend the hierarchy later. v0.0 stops at continent + country per the brief.
- **No automatic role assignment from `diaspora_status` text** (e.g., free-text `"returnee"` → enum `'returnee'`). The product call about overlap between `diaspora_status` ancestry tagging and D054 role is left open; documenting in the audit doc §1.

## Pre-flight checklist (when the migration is eventually authored)
